# NexusSwarm v2 — API Specification

The **NexusSwarm v2** backend exposes a robust REST API and WebSocket gateway built with FastAPI. It handles task submissions, multi-agent pipeline progress tracking, generated artifact retrieval, semantic memory searches, real-time logging, and dynamic model benchmarking.

For core code definitions, see [main.py](file:///c:/hack/backend/main.py) and [routes.py](file:///c:/hack/backend/routes.py).

---

## 1. Endpoints Overview

| Method | Endpoint | Description | Auth Required | Rate Limit |
| :--- | :--- | :--- | :---: | :---: |
| `POST` | `/auth/google` | Authenticates users using Google ID tokens. | No | 60/min |
| `GET` | `/agents` | Retrieves the list of available agents and their current model assignments. | Yes | 60/min |
| `POST` | `/submit-task` | Submits a task description, run security checks, and spawns the Swarm core. | Yes | 10/min |
| `GET` | `/tasks` | Lists all submitted tasks and their high-level completion status. | Yes | 60/min |
| `GET` | `/task/{task_id}` | Retrieves status, active pipelines, and details of a specific task. | Yes | 60/min |
| `GET` | `/status/{task_id}` | Alias for `/task/{task_id}` for polling task status. | Yes | 60/min |
| `GET` | `/task/{task_id}/outputs` | Retrieves the final generated outputs (code, test results, reports) for a task. | Yes | 60/min |
| `GET` | `/files/{task_id}` | Lists all files generated in the workspace directory for a specific task. | Yes | 60/min |
| `GET` | `/files/{task_id}/download` | Prepares and downloads a ZIP archive of all files generated by a task. | Yes | 30/min |
| `GET` | `/files/{task_id}/{filename}`| Downloads a specific code or documentation file generated by a task. | Yes | 60/min |
| `GET` | `/memory/search` | Performs a semantic text search across the agents' long-term memory. | Yes | 30/min |
| `GET` | `/models/performance` | Retrieves benchmarking and token performance history of models in the registry. | Yes | 60/min |
| `POST` | `/models/benchmark` | Runs an evaluation to find the optimal model choice for a task type. | Yes | 30/min |
| `GET` | `/stats` | Retrieves system-wide stats (total runs, success rates, average latency). | Yes | 60/min |
| `GET` | `/stats/leaderboard` | Returns performance leaderboard scores of various LLM providers (NIM/Azure). | Yes | 60/min |
| `POST` | `/stats/leaderboard/run` | Triggers a fresh run to recalculate leaderboard model performance. | Yes | 10/min |
| `GET` | `/health` | System health check (verifies database, Redis cache, and NIM/Azure connectivity).| No | None |
| `GET` | `/debug-file` | Helper endpoint to inspect debug and session files locally. | Yes | None |

## 2. API Security & Guardrails

API endpoints are secured at multiple layers:
1. **Adversarial Prompt Defense**: Handled by `check_adversarial_prompt()` inside [routes.py](file:///c:/hack/backend/routes.py#L175-L200). Blocks SQL injections, code executions (e.g. `__import__`, `eval`), prompt injection keywords (e.g. `"ignore previous instructions"`), and DoS file-bomb payloads (e.g. referencing >100 files or >60 packages).
2. **Token Budget Guard**: Handled in [security_utils.py](file:///c:/hack/backend/security_utils.py). Enforces an IP-based token quota per client to prevent spam/DoS attacks on expensive model backends.
3. **Rate Limiting**: Enforced via slowapi middleware with individual limits per route (e.g. `/submit-task` is capped at 10 requests per minute). Exceeding these returns a `429 Too Many Requests` response.

## 3. Core Schema & Payload Examples

### 3.1. `POST /submit-task`
Submits a task for the Swarm to decompose and build.

**Request Payload:**
```json
{
  "title": "Build a secure FastAPI task system",
  "description": "Create a FastAPI task manager backend with SQLite storage and basic unit tests.",
  "parent_task_id": null
}
```

**Response Payload (`TaskResponse`):**
```json
{
  "task_id": "4a8e3290-7d72-4e89-bd99-f2e1a3bc893a",
  "status": "pending",
  "message": "Task received and queued successfully."
}
```

### 3.2. `GET /agents`
Returns active model allocations for each agent role.

**Response Example:**
```json
{
  "provider": "nvidia",
  "agents": [
    {
      "name": "HeadOrchestrator",
      "level": "orchestrator",
      "model": "deepseek-ai/deepseek-v4-flash",
      "provider": "NVIDIA NIM"
    },
    {
      "name": "EngineeringManager",
      "level": "manager",
      "model": "qwen/qwen2.5-coder-32b-instruct",
      "provider": "NVIDIA NIM"
    },
    {
      "name": "BackendAgent",
      "level": "worker",
      "model": "nv-mistralai/mistral-nemo-12b-instruct",
      "provider": "NVIDIA NIM"
    }
  ]
}
```

### 3.3. `GET /task/{task_id}`
Returns the progress, active pipelines, and logs of a specific swarm run.

**Response Example:**
```json
{
  "id": "4a8e3290-7d72-4e89-bd99-f2e1a3bc893a",
  "title": "Build a secure FastAPI task system",
  "status": "engineering",
  "priority": 1,
  "created_at": "2026-07-18T13:00:00Z",
  "updated_at": "2026-07-18T13:01:30Z",
  "pipelines": [
    {
      "name": "planning",
      "status": "done",
      "progress": 100,
      "manager_agent": "PlanningManager"
    },
    {
      "name": "engineering",
      "status": "active",
      "progress": 45,
      "manager_agent": "EngineeringManager"
    }
  ]
}
```

## 4. Real-Time Streaming Gateway (`WEBSOCKET /ws/agents`)

To render active logs, node highlights, and agent status changes in the frontend canvas, the backend publishes events over Redis pub/sub. Web clients connect to the websocket endpoint `/ws/agents` to receive live JSON events.

### 4.1. Event Message Schema
```json
{
  "task_id": "4a8e3290-7d72-4e89-bd99-f2e1a3bc893a",
  "pipeline_name": "engineering",
  "agent_name": "BackendAgent",
  "agent_level": "worker",
  "event_type": "agent_action",
  "message": "Writing backend task endpoints inside main.py",
  "status": "in_progress",
  "payload": {
    "file_written": "backend/main.py",
    "lines_count": 142
  },
  "created_at": "2026-07-18T13:01:35Z"
}
```
